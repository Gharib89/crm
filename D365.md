# Dynamics 365 Customer Engagement (on-premises) v9.x — Agent CLI

## Overview

A stateful CLI harness for **Microsoft Dynamics 365 Customer Engagement on-premises, version 9.x**.
The CLI wraps the real **Dataverse Web API** (OData v4) exposed by the on-prem server at
`https://<server>/<org>/api/data/v9.x/`, using **NTLM** (Windows Integrated) authentication.

The "real software" in this case is the live D365 server. The CLI does **not**
reimplement CRM logic — every command issues a real HTTP request to the server
and returns the real response.

## Backend Engine

- **Web API endpoint**: `https://<host>/<org>/api/data/v9.x/`
- **Protocol**: OData v4 over HTTPS
- **Auth**: NTLM via `requests_ntlm.HttpNtlmAuth(f"{domain}\\{user}", password)`
- **Required headers**:
  - `OData-MaxVersion: 4.0`
  - `OData-Version: 4.0`
  - `Accept: application/json`
  - `Content-Type: application/json; charset=utf-8`
  - `If-None-Match: null` on `POST` to suppress upsert
  - `Prefer: return=representation` to get the created/updated record back
  - `Prefer: odata.include-annotations="*"` to get formatted values
- **Verbs**: `GET` (retrieve), `POST` (create), `PATCH` (update / upsert), `DELETE`, `PUT` (single property)

## GUI → API Mapping

| User-facing GUI action                  | Web API call                                                   |
|-----------------------------------------|----------------------------------------------------------------|
| Create record (e.g. account)            | `POST /accounts`                                               |
| Open record                             | `GET /accounts(<guid>)`                                        |
| Edit record (save)                      | `PATCH /accounts(<guid>)`                                      |
| Delete record                           | `DELETE /accounts(<guid>)`                                     |
| Advanced Find / saved query             | `?fetchXml=<urlencoded fetch>` or `$filter=`                   |
| Lookup picker                           | `$filter` + `$select`                                          |
| Solution Manager – export solution      | `ExportSolution` action                                        |
| Solution Manager – import solution      | `ImportSolution` action                                        |
| User identity probe                     | `WhoAmI` function                                              |
| View entity in Customizations           | `EntityDefinitions(LogicalName='<x>')`                         |
| View attribute properties               | `EntityDefinitions(LogicalName='<x>')/Attributes(LogicalName='<y>')` |
| Picklist values                         | `RetrieveAttributeRequest` / OptionSet metadata                |

## Data Model

- **Entity sets**: server-defined; logical name → entity set name (e.g. `account` → `accounts`,
  `contact` → `contacts`, custom `new_thing` → `new_things`). Always discoverable via
  `$metadata` or `EntityDefinitions`.
- **Primary key**: always a GUID returned in `OData-EntityId` response header.
- **Lookups**: navigation properties prefixed with `_<logical>_value` for raw GUID,
  or expand-able as nested objects.
- **OptionSets**: integer values with `@OData.Community.Display.V1.FormattedValue` annotations.

## Command Groups (CLI Surface)

| Group        | Commands                                                          | Purpose                                  |
|--------------|-------------------------------------------------------------------|------------------------------------------|
| `connection` | `connect`, `status`, `disconnect`, `whoami`, `test`, `profiles`   | Manage server connection + profiles      |
| `entity`     | `get`, `create`, `update`, `delete`, `upsert`, `list`             | Record CRUD                              |
| `query`      | `odata`, `fetchxml`, `count`                                      | Read queries (OData v4 or FetchXML)      |
| `metadata`   | `entities`, `entity`, `attributes`, `attribute`, `relationships`  | Schema introspection                     |
| `solution`   | `list`, `info`, `export`, `import`, `components`                  | Solution lifecycle                       |
| `data`       | `export`, `import`                                                | Bulk dataset CSV/JSON I/O                |
| `action`     | `function`, `invoke`                                              | Call bound/unbound OData functions/actions |
| `session`    | `info`, `save`, `clear`, `history`                                | Local session state                      |

All commands support `--json` for machine-readable output.
REPL is the default entry point when `crm` is run with no subcommand.

## State Model

The CLI is stateful for **convenience**, not for mutation. Each mutation hits the server
immediately (no local change buffer).

State persisted on disk (`~/.crm/`, override with `CRM_HOME`):

- `profiles/<profile>.json` — connection profile: url, domain, user, org_name, api_version.
- `sessions/<session>.json` — current session: active profile, last entity context, last query.
- `history` — prompt_toolkit history file.

Credentials policy:
- **Never** persist passwords on disk.
- Read from env: `D365_URL`, `D365_DOMAIN`, `D365_USERNAME`, `D365_PASSWORD`, `D365_AUTH=ntlm`.
- Optional `--password` flag for explicit override (discouraged for production).

## Auto-save & --dry-run

D365 is **not** a project-file editor — there is no "open project, modify, save" cycle.
Each mutation is a single HTTP call to the server. Therefore the auto-save-on-exit pattern
documented in `guides/auto-save-dry-run.md` does **not** apply here.

However, `--dry-run` is still honored: when set, mutation commands print the would-be HTTP
request (method, URL, body) instead of issuing it. This lets agents preview server changes
before committing.

Session state (active profile, last entity, last query) is auto-saved after every
successful command so subsequent REPL/one-shot invocations resume context.

## Output Modes

- **Human**: tables, colored status lines, key/value blocks via `ReplSkin`.
- **JSON** (`--json`): single JSON document on stdout, structured as
  `{ "ok": bool, "data": ..., "error": ..., "meta": {...} }`.

## Required Dependencies

- Python ≥ 3.9
- `click ≥ 8.0` — command framework
- `requests ≥ 2.28` — HTTP
- `requests_ntlm ≥ 1.2` — NTLM auth (Windows Integrated)
- `prompt_toolkit ≥ 3.0` — REPL
- A reachable D365 on-prem 9.x server is a **hard runtime dependency**. Without it the
  CLI has nothing to talk to. E2E tests fail loudly if `D365_URL` is unset.

## Verification Strategy

- **Unit tests** (`test_core.py`): mock the HTTP layer via `responses`/`requests_mock`,
  exercise URL building, payload shape, header policy, OData filter compilation, FetchXML
  URL encoding, session serialization.
- **E2E tests** (`test_full_e2e.py`): require `D365_URL`/`D365_USERNAME`/`D365_PASSWORD`
  env vars. Run a real CRUD round-trip against the live server, including `WhoAmI`,
  contact create → update → delete, and a FetchXML query. Fail loudly if env is missing.
- **CLI subprocess tests**: run the installed `crm` via `_resolve_cli()`,
  exercise `--help`, `--json connection status`, `--json metadata entities`, and a
  full create-read-delete round-trip.
