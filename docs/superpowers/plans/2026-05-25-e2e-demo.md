```markdown
# CRM CLI

**Stateful CLI for Microsoft Dynamics 365 Customer Engagement on-premises, version 9.x.**
Wraps the Dataverse Web API (OData v4) over HTTP/S with NTLM (Windows Integrated) authentication.

---

## Why

D365 CE on-prem ships with a GUI and a .NET SDK. There is no first-party CLI optimised for
shell scripting or AI agents. This harness gives you:

- `crm` in your PATH
- One-shot subcommands for record CRUD, OData and FetchXML queries, metadata introspection, and solution lifecycle
- A stateful REPL for ad-hoc admin work
- `--json` everywhere for agent / pipeline consumption
- `--dry-run` to preview any HTTP request before issuing it

## Requirements

| Requirement | Version | Notes |
|-------------|---------|-------|
| Python | ≥ 3.9 | |
| D365 CE on-prem | 9.0 / 9.1 / 9.2 | Reachable over HTTP/S |
| Auth | NTLM | OAuth/IFD out of scope |

## Install

```bash
pip install -e .
crm --version
```

## Configure

Credentials come from environment variables or a saved profile.

```bash
export CRM_BASE_URL="https://crm.corp/MyOrg"
export CRM_USERNAME="DOMAIN\\user"
export CRM_PASSWORD="..."
export CRM_DOMAIN="CORP"        # optional — parsed from username if omitted
export CRM_AUTH="ntlm"          # default
export CRM_API_VERSION="v9.2"   # default v9.2
```

Or use the interactive wizard:

```bash
crm init
```

## Quick start

```bash
crm connection whoami
crm query odata accounts --select name,telephone1 --top 10
crm --json entity create contacts --data '{"firstname":"Ada","lastname":"Lovelace"}'
crm data export contacts -o contacts.csv --select fullname,emailaddress1
crm   # interactive REPL
```

## Command groups

| Group | Purpose |
|-------|---------|
| `connection` | Profiles, WhoAmI, reachability checks |
| `entity` | Record CRUD (get / create / update / upsert / delete) |
| `query` | OData v4 and FetchXML queries |
| `metadata` | Entity / attribute / relationship / option set CRUD |
| `solution` | List / info / components / export / import / publish |
| `data` | Bulk CSV / JSON dataset export |
| `workflow` | List, activate, and trigger D365 workflows |
| `action` | Call arbitrary OData functions and actions |
| `batch` | Execute an OData `$batch` request from a JSON file |
| `async` | List, inspect, and cancel async operations |
| `session` | Local session state and command history |
| `repl` | Interactive REPL with tab completion |
| `init` | Bootstrap workspace (.env template + profile wizard) |

Use `crm <group> --help` for details.
```

# batch

Execute an OData `$batch` request from a JSON file.

**Syntax:** `crm batch FILE_PATH [OPTIONS]`

| Flag | Type | Description |
|------|------|-------------|
| `--no-transaction` | flag | Send as top-level operations (no changeset) |
| `--continue-on-error` | flag | `Prefer: odata.continue-on-error` (requires `--no-transaction`) |
| `--output` | FILE | Write `BatchResult[]` JSON to this path |
| `--timeout` | INTEGER | Override request timeout (seconds) |

## Input format

```json
[
  {"method": "POST", "url": "accounts", "body": {"name": "Contoso"}},
  {"method": "PATCH", "url": "accounts(<guid>)", "body": {"telephone1": "555-0100"}},
  {"method": "DELETE", "url": "accounts(<guid>)"}
]
```

`url` is relative to the API base. Omit `body` for DELETE / GET.

## Example

```bash
crm --json batch docs/demo/batch-create-tickets.json
# → {"ok": true, "data": [{"index": 0, "status": 204}, ...], "meta": {}}
```

!!! note
    By default all operations are wrapped in a changeset — all succeed or all fail.
    Use `--no-transaction` for mixed read/write batches.
