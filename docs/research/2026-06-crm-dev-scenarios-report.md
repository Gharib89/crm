# CRM Developer Scenarios — Research Synthesis Report (June 2026)

**Date:** 2026-06-10
**CLI assessed:** `crm` v2.4.0 (release binary)
**Trial org:** `crmworx` — local on-prem D365 CE v9.1 test environment (full-mutation sandbox)
**Inputs:** six harvest files (`harvest/2026-06/00-prior-art.md`…`05-ai-assisted.md`, 154 rows), the [scenario catalogue](2026-06-crm-dev-scenario-catalogue.md) (59 scenarios), the [coverage matrix](2026-06-coverage-matrix.md), the [skill-trial plan](2026-06-skill-trial-plan.md), and the [skill-trial log](2026-06-skill-trial-log.md).

This report synthesises four research phases (harvest → catalogue → coverage matrix → hands-on skill trial) into findings and a recommendation backlog. The next task drafts a 1:1 GitHub issue list from Section 6.

## 1. Executive summary

**What we did.** We harvested 154 task signals from six independent source families (prior-art research docs, Microsoft Learn, community/Stack Overflow, the XrmToolBox plugin catalogue, the `pac` CLI command surface, and AI-assisted-dev posts), merged them into a 59-scenario catalogue across seven domains, classified every scenario against the live `crm` v2.4.0 command tree on two axes (can the CLI do it; does the skill teach it), then ran eight hands-on agent trials against a real on-prem v9.1 org to flush skill gaps and CLI bugs.

**Top findings.**

1. **The CLI's coverage is strong: 35 of 59 scenarios (59%) are fully covered** — CLI capability plus skill guidance. The agentic-ergonomics surface the prior-art catalogue asked for (declarative `apply`, `metadata describe` write-readiness brief, `meta.category` error taxonomy, `--expect`/`--validate`, dry-run, session audit) is largely shipped and taught.
2. **The skill text is accurate, not stale.** Across SKILL.md and all 9 reference files, zero wrong/stale claims were found against v2.4.0. The 3 skill-gaps (SCN-011, SCN-052, SCN-053) are all *absences* — first-class commands that exist but no reference file names them.
3. **The on-prem-only cliff is the strongest signal for new CLI work.** Of 20 cli-gaps, five (SCN-017, 022, 027, 037, 054) have *only* an online-only external tool (`pac`, Command Designer, Power Automate, Solution Layers Explorer) — meaning on-prem developers have *no* scripted path at all. Most other gaps have a dual-target external tool (XrmToolBox/PRT via the SDK) that the CLI could wrap or simply document.
4. **Failures cluster after confusing server responses, not on the happy path.** Both severe trial failures (TRIAL-5 escalation flailing, TRIAL-8 fabricated diagnosis) happened *after* a confusing server response. Error-path clarity matters more than happy-path docs for unattended agents.
5. **Three confirmed CLI bugs and one structural on-prem evidence hole** surfaced from the trials: `metadata get-optionset` 400s on v9.1, malformed `--data` crashes with a raw traceback (envelope-contract violation), `solution remove-component` sends a parameter v9.1 rejects, and CLI imports never create an `ImportJob` row on-prem (so `import-result` is structurally unusable there).

**Top 5 recommendations** (full detail + evidence in Section 6):

1. **Fix `metadata get-optionset` on v9.1** (B1) — the dedicated read verb is unusable against the on-prem target while the generic path works; cast to the derived type.
2. **Wrap `_load_payload` JSON parsing in the envelope contract** (B2) — malformed `--data` currently emits a raw Python traceback and breaks the `{ok:false}` contract every agent relies on.
3. **Fix `solution remove-component` add/remove asymmetry on v9.1** (B3) — send the `SolutionComponent` entity-reference shape the workaround uses.
4. **Teach the three first-class-but-unmentioned commands in the skill** — `solution import-result`/`job-status` (SCN-053), the `plugintracelogs` trace-debug loop (SCN-052), and the edit-existing-view savedquery PATCH recipe (SCN-011).
5. **Close the on-prem import-verification hole** — the highest-value *build*: pass/generate an `ImportJobId` on import so `import-result` has something to fetch, and add a one-line skill router mapping "list/query records → `query odata`" to erase the recurring verb-guess class seen in 3 of 8 trials. (The prior candidate, a scriptable workflow/process inventory for on-prem, is already covered.)

## 2. Personas and day-in-the-life workflows

Two personas were selected for this research (the spec's "Decisions" table; ALM/DevOps and support/ops personas were noted in passing only). Workflow claims below are grounded in the harvest source families.

### Pro-code developer (plugins, JS/web resources, ribbon, queries, integrations)

A typical day is a tight server-side or client-side iteration loop:

- **Write → register → trigger → read trace logs.** Plugin work is the primary code artifact on every project and recurs multiple times per sprint, with no hot-reload — compile, register a step, trigger it, then read trace logs to see what happened (xrmtoolbox: `https://www.solvingdynamics365.com/guides/plug-in-registration-tool-deep-dive`; ai-assisted: `https://www.xrmtoolbox.com/plugins/Cinteros.XrmToolBox.PluginTraceViewer/`). Wrong stage/image names fail silently. The Plugin Trace Viewer's 608k+ lifetime downloads are a direct proxy for how often this loop runs and how unusable the native trace UI is at volume (xrmtoolbox: `https://crmcurator.com/articles/dynamics365/plugin-trace-log-analysis/`).
- **Compose FetchXML/OData, run it, embed it.** The query layer sits under every view, report, plugin retrieval, and integration. FetchXML Builder is the #1 XrmToolBox plugin by total downloads (1.06M), evidencing how routine — and how painful in raw XML — this is (community+xrmtoolbox: `https://www.xrmtoolbox.com/plugins/Cinteros.Xrm.FetchXmlBuilder/`). Teams hit 5,000-row paging and 50,000-row aggregate caps and 30-second timeouts on unindexed columns (community: `https://dev.to/sapotacorp/dataverse-web-api-fetchxml-vs-odata-patterns-that-dont-time-out-2e6p`).
- **Upload → publish → cache-clear for web resources.** Every JS/HTML/CSS change is an upload-then-publish cycle, several per hour, with a hard browser-cache refresh and no hot-reload (community: `https://dev.to/kkazala/client-script-in-model-driven-apps-do-it-right-from-the-start-19c5`; ai-assisted: `https://github.com/ImranAkram/XrmToolBox.AutoDeployer`).
- **Authenticate and integrate over the Web API.** Every scripted session and integration must authenticate; on-prem has no first-party .NET 6+ NTLM path (community: `https://github.com/janis-veinbergs/DataverseServiceClientOnPremSamples`), and online integrations drown in AADSTS/scope/app-user errors — the top Stack Overflow cluster (community: `https://stackoverflow.com/questions/71968454/dynamics-365-oauth-2-0-authentication-failed`). `@odata.bind` nav-property casing must match `$metadata` exactly (mslearn: `https://learn.microsoft.com/power-apps/developer/data-platform/webapi/overview`).
- **Increasingly: drive the whole loop from an AI agent.** Every first-party Dataverse AI tool is Entra/cloud-only, so the viable on-prem path is a generic coding agent orchestrating an NTLM-aware CLI bridge — the exact role `crm` fills (prior-art: `ai-agent-tooling-onprem.md`).

### Customizer / functional consultant (tables, forms, views, BPFs, roles, apps)

A typical engagement is declarative configuration moving through environments:

- **Stand up the data model.** Create custom tables with a primary-name column and 1:N/N:N relationships — foundational to every implementation; the main friction is naming discipline (immutable logical names) and remembering to publish (mslearn: `https://learn.microsoft.com/power-apps/maker/data-platform/fields-overview`). Schema is iterated dozens of times across a project (prior-art: `onprem-automation.md`).
- **Author forms and views.** Every new table or requirement touches form layout (designer-only, publish easy to forget) (mslearn: `https://learn.microsoft.com/dynamics365/customerengagement/on-premises/customize/create-edit-main-forms?view=op-9-1`); views are customised on almost every entity, and editing an existing saved view scriptably has no clean path (community: `https://community.dynamics.com/blogs/post/?postid=ef44c4f2-0e6a-45ac-8464-5a8bdd3312cc`).
- **Author declarative processes.** Business process flows are standard deliverables on virtually every Sales/Service implementation; BPF/classic-workflow/business-rule authoring spans three disjoint designer-only surfaces with no supported API path (mslearn: `https://learn.microsoft.com/power-automate/create-business-process-flow`).
- **Configure security and apps.** New apps and tables are invisible until roles are assigned; the privilege grid is 200+ rows × 8 access levels and the native UI is one-user-at-a-time (mslearn: `https://learn.microsoft.com/power-platform/admin/database-security`). Roles must travel across environments inside solutions, where managed import silently strips manually-added privileges (community: `https://github.com/MicrosoftDocs/power-platform/blob/main/power-platform/alm/how-managed-solutions-merged.md`).
- **Promote through dev → test → prod.** Every feature delivery moves through the environment chain; ~45–50% of first manual import attempts fail on missing dependencies, and unmanaged-layer drift silently blocks managed updates (community: `https://www.solvingdynamics365.com/guides/solution-import-export-pipelines`; community: `https://dev.to/sapotacorp/solution-layering-in-dataverse-why-managed-changes-dont-appear-bmj`).

### 6A. Skill enhancements

Project rule: the skill states only what `crm describe`/`--help` cannot — never restate flags/choices/defaults. Recipes, workflows, and gotchas are fine; new `docs/how-to/` pages are allowed. Every item below adds a workflow/gotcha, not a flag list.

## Methodology lessons (coda)

Two lessons for future agent-driven research, both earned from this run:

1. **Envelope parsing must check `ok:` first.** The phantom `workflow list --category` "bug" was manufactured by a controller probe wrapper that read `data`/`meta.count` without checking `ok:` — a clean exit-2 usage error was misread as "0 rows," inventing a CLI bug that did not exist (TRIAL-5; reclassified in the NOT-A-BUG repro). Any harness consuming `crm --json` output must branch on `ok:` before touching `data`.
2. **Trial premises need a contradiction check.** TRIAL-8's premise ("this zip fails to import") was false because the controller's export auto-included the required parent entity. The agent never reconciled `ok:true, status:succeeded` against "this file fails to import," and instead fabricated a confident, internally-coherent, wrong root cause from collisions its own import had just created. Both agent and controller must validate that a task's stated premise actually holds before acting on it — and (cross-trial pattern 3) transcript review, not agent self-report, is the only reliable evaluation signal.
