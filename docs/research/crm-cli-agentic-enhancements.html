Enhancing the crm CLI for agentic coding & D365 on-prem customization

A repo-grounded, adversarially verified set of enhancements: how to make the Dynamics 365 CE on-prem (v9.x, Web API / NTLM) CLI a first-class actuator for AI coding agents and for on-prem solution / customization ALM.

generated 2026-06-03 D365 CE on-prem v9.x NTLM · Web API OData v4 verified vs the repo

Overview

crm is a Python Click CLI that drives a Dynamics 365 CE on-premises (v9.x) server over the Dataverse Web API (OData v4) with NTLM auth, and it is already the only viable agentic bridge to on-prem: it ships a stable {ok,data,error,meta} JSON envelope, a committed 0/1/2 exit-code contract, --dry-run request previews, --stage-only batch-then-publish, a two-layer destructive-op gate, and a bundled SKILL.md so a Claude Code or Copilot agent can shell out and parse stdout. The single biggest leverage area is the agent OUTPUT CONTRACT plus DISCOVERY: small, surgical envelope and introspection changes (normalize meta.dry_run, add a machine-readable error taxonomy, expose a describe --json command tree, add meta.warnings[]) materially de-risk autonomous loops at S effort, and they unblock everything else. The second thrust is closing the ALM authoring loop for agents: a one-shot metadata describe write-readiness brief, declarative apply -f spec, delete-attribute/delete-relationship to match the create verbs, ImportJob per-component verification, solution version bump, and SolutionPackager extract/pack for source-control diff. MCP work (a thin crm mcp serve plus a portable destructive gate) is real and is the only way to give Copilot schema'd tools, but it reverses a documented "drop MCP" decision and should be gated behind an explicit Copilot-parity commitment. The recommended sequencing: ship the cheap contract/discovery wins now, then the high-impact author/verify ALM primitives, then the connection-bound and MCP investments.

Current state

crm is a single-package Python Click CLI targeting D365 CE on-prem v9.x exclusively (NTLM only; OAuth/IFD/Claims out of scope), talking to the Dataverse Web API through crm/utils/d365_backend.py with a mockable backend. It already implements the core agent-safety guardrails (JSON envelope, exit-code contract, dry-run, stage-only, destructive gate, lazy command loading) and a broad customization surface (entity/attribute/optionset/relationship CRUD, views, apps, sitemaps, publisher/solution create, solution import/export/publish). The gaps cluster in three areas: agent output-contract/discovery ergonomics, ALM authoring completeness (deletes, version bump, component add/remove, web resources, plugins, sitemap builder), and the absence of any MCP surface or declarative apply model.

Strengths

  • Stable {ok,data,error,meta} JSON envelope on every command via a single CLIContext.emit chokepoint (cli.py:47-61), recursively sanitized to be JSON-serializable
  • Committed 0/1/2 exit-code contract (ADR 0001): emit(ok=false) raises Exit(1) so failures never silently exit 0; usage errors stay 2 and are JSON-wrapped under --json
  • --dry-run previews the exact HTTP request without issuing it (d365_backend.py:305-313), with zero side effects guaranteed by ADR 0002
  • --stage-only forces batch-then-publish-once, sticky in the REPL, with staged:true recorded in meta
  • Two-layer destructive gate: in-band --yes/_confirm_destructive plus a deterministic Claude Code PreToolUse hook that hard-blocks ungated destructive Bash calls and resists shell evasion
  • Lazy command loading keeps cold start fast for one-shot agent invocations; _lazy_commands is the single registration point
  • Broad on-prem-correct customization surface: 14 attribute kinds, retrieve-merge-write metadata PUT with MergeLabels, global optionset @odata.bind MetadataId resolution, 1:N/N:N relationships, public views, model-driven apps, solution import/export with ImportJob polling
  • Bundled installable SKILL.md instructing agents to pass --json, branch on exit code, dry-run before mutations, and pass --yes non-interactively
  • Zero-touch profile auto-wiring (ADR 0002) writes publisher_prefix/default_solution back into the active profile; passwords never persisted

Gaps

  • No machine-readable command/capability discovery: no describe --json, no --help --json; the _lazy_commands registry is private and SKILL.md prose drifts (already 15 commands stale)
  • Dry-run signal is an undocumented _dry_run sentinel buried in data with no stable meta.dry_run flag; shape varies per command and batch/poll paths carry no sentinel at all
  • Error envelope carries only raw meta.status/meta.code; no enumerable failure category or retryable hint, so every agent re-encodes the same hex-to-meaning playbook from prose
  • No structured warnings channel: singular meta.warning clobbers; optionset partial-failure context (completed_steps/stage) is dropped at the command boundary
  • Bare crm with no subcommand drops into the interactive REPL with no CRM_NO_REPL/non-TTY guard, so an agent omitting a verb hangs or misreads exit 0
  • ALM authoring asymmetry: can create attributes/relationships but cannot delete them; no solution version bump, no AddSolutionComponent/RemoveSolutionComponent, no web resource or plugin registration, no structured sitemap builder
  • No declarative desired-state apply and no spec export; every authoring action is a separate imperative subprocess the agent must order correctly
  • solution import overwrites unmanaged customizations by default and is not in the destructive gate; ImportJob results are not parsed into per-component pass/fail
  • No bulk data import (inverse of data export), no MCP server, no persistent metadata cache, no connection doctor preflight

Output contract & agent ergonomics

#1

Normalize the dry-run signal to a stable meta.dry_run flag

Output contract & agent ergonomicsimpactMediumeffortSmallR1
Problem

A dry-run returns a raw {_dry_run, method, url, params, headers, body} dict as data with no top-level or meta-level signal; the shape varies per command (metadata update adds diff/path, create ops add _exists/would_skip) and batch/poll previews carry no _dry_run key at all. An agent must sniff a magic key inside data, and docs/index.html:649 already advertises meta.dry_run as a feature that does not exist.

Proposal

In CLIContext.emit (cli.py:47), when self.json_mode and self.dry_run, set meta['dry_run']=True before building the envelope, merging into existing meta (e.g. {'staged':True}) without clobbering. Key it off the invocation-level ctx.dry_run flag (cli.py:310), NOT off sniffing data, so batch (list-shaped) and poll previews are covered uniformly and forced-real existence-probe GETs do not false-positive. Keep the in-data _dry_run sentinel for back-compat (ADR 0002 guards read it pre-emit). Scope to JSON mode. Document meta.dry_run in SKILL.md as the canonical signal. Add tests: meta.dry_run on an entity-create dry-run and on a batch dry-run.

Why it matters

Single-line change at the one emit chokepoint that uniformly normalizes every command, fixes a live documentation lie, and is the prerequisite an MCP layer would model dry_run on. ADR 0001 explicitly endorses cross-cutting signals in meta.

#2

Add a machine-readable error taxonomy: meta.category + meta.retryable

Output contract & agent ergonomicsimpactMediumeffortSmallR2
Problem

_handle_d365_error (_helpers.py:31-35) emits only meta={status, code}; code is the raw server hex (e.g. 0x80040217) plus two fragile hand-synthesized strings (412->PreconditionFailed, 403->MissingPrivilege keyed on a message substring). There is no enumerable failure category an agent can switch on and no retryable hint. The recovery playbook lives only as prose (SKILL.md:357-361), forcing every agent to re-encode the same hex-to-meaning mapping; the transport-failure path raises with status=None AND code=None, indistinguishable from anything else.

Proposal

Add a pure classify_d365_error(status, code, message)->(category, retryable) helper near D365Error in d365_backend.py and have _handle_d365_error additively set meta.category and meta.retryable (keep status/code). Closed enum: not_found, auth_failed, forbidden, concurrency_conflict, duplicate_detected, validation, throttled, server_error, transport_error. Map status-first: None->transport_error(retryable), 401->auth_failed, 403->forbidden (subsuming the fragile MissingPrivilege synthesis), 404/0x80040217->not_found, 412->concurrency_conflict(retryable), 429->throttled(retryable), 0x80040237/DuplicateRecord->duplicate_detected, 5xx->server_error(retryable), other 400->validation. retryable=true only for transient classes; frame it as a post-exhaustion class hint (the backend already auto-retries transient failures), with concurrency_conflict meaning refetch-then-retry. Replace the prose list in BOTH SKILL.md copies with the enum table. Add a mapping test.

Why it matters

ADR 0001 chose flat exit codes precisely because failure-class granularity should live in the envelope, yet the envelope has no class field. This closes a self-inflicted gap at S effort, is more robust than the existing substring synthesis, and gives the only machine signal on the status-less transport path.

Problem

The partial-failure sub-change recovers currently-dropped context on the single deliberate non-atomic multi-write in the codebase, letting an agent drive a deterministic compensating loop without re-querying. #25

Add a client-side retry-suppression guard for non-idempotent POST creates

Output contract & agent ergonomicsimpactMediumeffortSmallR25
Problem

entity create has no idempotency guarantee. The transport retry loop auto-retries 429/5xx and transient transport errors regardless of HTTP method, so a POST create that succeeds server-side but loses its response to a transport failure can be silently re-issued, creating a DUPLICATE. For an agent looping on exit codes that may re-run a create after an ambiguous failure, there is no dedup guarantee (D365 on-prem 9.x has no server-honored client idempotency key).

Proposal

Make the automatic retry loop NOT auto-retry non-idempotent POST record-creates by default. Both changes are required to close the window: (1) gate the transport-exception branch (d365_backend.py:328-336) to skip retry when method=='POST' unless an explicit opt-in says the caller accepts re-send risk; (2) ALSO stop _is_response_retryable from auto-retrying POST on 429/503 for record creates — otherwise a 503-after-commit still duplicates. Preserve current POST-503/429 retry only for $batch (already documented as caller's responsibility). Add an opt-in --retry-on-ambiguous (default off). Document create-retry semantics in SKILL.md: a POST whose response is lost may have committed — re-running risks a duplicate; use entity upsert (by id) for idempotent writes. Soften upsert-by-alternate-key as a general fix (most entities have no alt key). Coordinate with issue #18.

Why it matters

Closes a narrow but real footgun for looping agents. Client-side retry suppression (not a token), S effort for the gate + docs; the by-id upsert escape hatch already exists.

Add a non-interactive REPL guard (CRM_NO_REPL / non-TTY / --json) so agents fail fast

Discovery & self-descriptionimpactMediumeffortSmallR5
Problem

A bare crm with no subcommand unconditionally invokes the interactive REPL (cli.py:326-328, invoke_without_command=True). An agent that omits the verb — or whose argv is mangled — either hangs on an open-but-silent stdin or, on EOF stdin, exits 0 (the REPL goodbye), misreading a usage error as success and violating ADR 0001. Even bare crm --json falls into the REPL today. The CLI provides no guard; SKILL.md only warns in prose.

Proposal

At cli.py:326, before invoking the REPL, when invoked_subcommand is None: if json_mode (already a bound callback param) OR CRM_NO_REPL is truthy OR not sys.stdin.isatty(), emit the standard usage signal and exit 2 — under json_mode call _emit_usage_envelope("no subcommand given; run crm --help to list commands") then raise Exit(2); otherwise click.echo the error to stderr and raise Exit(2). Point the message at crm --help (which exists today), not a describe command unless R3 ships first. Explicit crm repl is unaffected (it only fires when invoked_subcommand is None). This is a proactive isatty probe, intentionally stronger than the EOF-driven _confirm_destructive. Add tests for the --json and CRM_NO_REPL branches (CliRunner reports isatty False), and a monkeypatched-TTY test asserting the REPL still launches for an interactive human.

Why it matters

Closes a real hang/wrong-exit-code failure mode on the discovery surface using existing plumbing (_emit_usage_envelope, Exit(2)). Pure CLI-entry guard, orthogonal to NTLM/on-prem. A backstop for misuse rather than a daily-path fix, hence Medium.

Add a persistent on-disk schema/metadata cache for one-shot agent calls

Discovery & self-descriptionimpactMediumeffortMediumR30
Problem

The only metadata cache is the in-memory REPL-session MetadataCache (repl.py:29-53). An agent driving the CLI as one-shot shell invocations (the documented primary architecture) gets zero caching: every metadata entities/describe re-fetches the full EntityDefinitions collection over NTLM, which does not support server-side $top so the whole list comes back each time. On-prem metadata throttling under NTLM is an open question, so repeated full pulls are slow and risky.

Proposal

Add an opt-in persistent read-only schema cache. (1) crm/core/metadata_cache.py stores under <CRM_HOME or ~/.crm>/cache/<profile>/entitydefs.json the {logical,set_name} list, embedding {url, api_version, cached_at} and treating a mismatch as a miss; reuse session._state_root() and _atomic_write_json.
